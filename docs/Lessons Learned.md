# Lessons Learned

**Status:** Phase 10 retrospective, covering Phases 1–9

## What went well

**Server authority, decided early, kept paying off.** The Phase 4 decision to make the server the sole authority for position, physics, score, and win conditions happened before any security-specific phase had even started, and it turned out to be the single highest-leverage decision in the whole project. It's why the Phase 5 "can the client cheat?" audit had fast, confident answers instead of a long list of open questions. It's also why Phase 9's adversarial testing found nothing to exploit in that category at all. Good architecture decisions made for ordinary engineering reasons often end up being good security decisions too; in practice, I couldn't really separate the two.

**Treating documentation as something the codebase gets checked against, not narrated for.** The Risk Register and Security Requirements were written by looking at what the code actually does and asking "does this hold?" rather than writing aspirational documentation first. This is why Software Requirements.md and Security Requirements.md both ended their own write-up with a "gaps found" section: writing the requirement *after* the system existed made it possible to notice what was never actually specified (concurrent room limits, token-in-transit protection). That's a real and useful failure mode to have caught on paper, rather than never.

**Distinguishing verification from adversarial testing.** This is the project's clearest lesson, not just a phase outcome. Phase 8 confirmed every control did what it was designed to do. Phase 9, a genuinely different exercise and not a repeat of Phase 8 with extra steps, found two severe bugs (R11, R12) that Phase 8's cooperative tests never could have found, because they were built to test the scenario the control was designed for, not the scenarios adjacent to it.

## What was challenging

**Keeping the phased plan honest under pressure to just fix things.** When R1 turned up in Phase 5, and again when R11/R12 turned up in Phase 9, the instinct was to fix immediately. Twice, that was the right call (SD-001, SD-002, SD-003 all document why). But that instinct needed a real justification each time (severity, likelihood, cost), not just impatience, or the phased structure would have quietly collapsed into "fix everything the moment it's found," which defeats the point of having phases at all. Writing the Security Decisions log was as much about disciplining that instinct as it was about producing a portfolio artifact.

**Server code stayed a single file longer than felt comfortable.** The Component Diagram in Phase 6 named this explicitly rather than fixing it immediately, on the reasoning that splitting `server.js` into modules would pay off more once there were more distinct concerns to separate (Phase 8's controls). In hindsight, that reasoning held. By Phase 9, the file had grown enough that a future pass genuinely would benefit from separating the room manager, physics engine, and socket handlers into their own modules. That refactor is now a clear, justified next step rather than a vague "someday" item.

**Reconnection logic was more subtle to get right than it first appeared.** The naive version (null the slot immediately on disconnect) was simple and wrong for anything beyond a hard crash: a brief WiFi drop would end an otherwise-fine match. The grace-period-plus-token version is correct but has more moving parts (pending-disconnect timers, token matching, room state that has to stay coherent across a socket swap), and it directly created R12 (the ghost-slot bug) as a side effect of that added complexity. More capability came with more surface area to get wrong, which is worth remembering next time "just add a grace period" sounds like a small change.

## What would be done differently

- **Write Security Requirements *before* Phase 5, not after Phase 9.** Doing it retrospectively in Phase 10 was useful for finding gaps, but a real project would want those requirements guiding implementation from the start, not just checking it afterward.
- **Add adversarial "what if this event fires with a malformed payload" testing as a standing checklist for every new socket event**, rather than something that only happened once, comprehensively, in Phase 9. R11 existed from the moment the `join` handler was first written in Phase 2; it sat undetected through Phases 3, 4, 5, 6, 7, and 8 before Phase 9 found it. Any handler added after this point should get that check immediately, not wait for a dedicated testing phase to come back around.

## Takeaways for the broader goal

This project's stated purpose was never really "build a multiplayer game." It was to generate honest, defensible evidence of security engineering judgement: architecture decisions with stated trade-offs, a risk register that actually gets revised as new information shows up, a threat model that formalizes earlier findings with STRIDE rather than throwing them out, and a testing phase that found real bugs instead of just confirming what I already believed. The R11/R12 story specifically is the kind of thing that's more convincing in an interview than a project with zero bugs ever could be. It shows the process actually working, not just the output looking clean.

## Post-deployment findings (after Phase 11)

Real user testing against the live deployment, not scripted tests, found three more issues. Worth recording those honestly rather than treating Phase 9 as having caught everything:

- **A genuine freeze bug**, found by actually playing a full match to completion: when the timer hit zero, the server correctly computed the winner internally but then skipped broadcasting that exact state, so the client never learned the match had ended and just froze on "1s" forever. Every automated test up to this point checked the *mechanics* of a match (join, physics, sync) but none had been run to a natural, timer-expiry conclusion. That's a gap in test coverage, not in the code review that came before it.
- **Two feature gaps** (no way to return to the mode-select screen without reloading, and no timer duration option for online matches) that were design omissions, not bugs. Reasonable to have deferred earlier, unreasonable to have left undiscovered until a real player hit them.
- **A reported "falling through a platform" glitch** that a targeted simulation (two bots, real physics ticks, climbing to 73m) could not reproduce as a server-side bug. The initial hypothesis, addressed by adding client-side position smoothing, was that it was a rendering artifact of zero interpolation over a real network connection with real latency, not an actual collision failure. That was a reasonable theory, backed by the best evidence available at the time, and it was wrong, or at least incomplete: the glitch kept happening after the smoothing fix shipped. The actual cause, found once the report persisted, was that the online input handler sent a network packet on every browser key-repeat event, not just the initial press, unlike Local mode's jump handling, which already filtered repeats correctly. Holding a key down could flood the rate limiter, which silently drops excess events; if an actual jump attempt got caught in that drop, the player just kept falling, with the server's own physics never having anything to get wrong. The bot simulation that found nothing at 73m couldn't have caught this either, for the same reason as the freeze bug above: it sent one input message on a fixed interval, never the bursty, repeat-heavy pattern a real held key produces. Worth recording plainly: the first, reasonable-sounding theory for a bug is not automatically the right one. "I couldn't reproduce this as described, here's my best theory and a real improvement regardless" is a fine thing to say once. It stops being fine if the report comes back a second time and gets the same answer instead of a harder look.
- **The platform pattern wasn't actually randomized per match.** `seedPattern()` ran once at server startup, and every new room copied that same never-reseeded array, meaning every online match, for roughly the first 40-70m, had an *identical* layout to every other match run against that same live server process. This is a genuine fairness concern, not just a cosmetic one (Threat Model.md lists game fairness as an asset worth protecting): a player who noticed the pattern could effectively memorize the early climb across repeated matches. Confirmed by code inspection alone (no live reproduction needed once the bug was understood), then verified fixed by creating three separate rooms in one running server process and confirming their initial layouts now differ.
- **Restart mid-match didn't actually restart anything.** Clicking Restart during an active online match reused the exact same disconnect-and-reconnect-with-token flow built for handling a dropped connection, which the server correctly interpreted as "resume the paused match," not "start a new one," leaving players in the same spot with the same platforms and just a confusing "disconnected" message. This was a case of one mechanism being asked to serve two different intents (recovering from an accident vs. a deliberate reset) that needed to be told apart explicitly. Fixed by adding a distinct, explicit restart event that resets the room in place without touching the connection at all. Reusing the reconnect flow for both cases was the actual mistake, not a bug in either flow individually.

A second round of real user testing, after the first round of fixes above had already shipped, found both of the last two issues. Worth being honest about what that means: fixing reported bugs doesn't guarantee the fix addresses the right root cause, and it doesn't mean testing is done. It means it's time to test the fix specifically, which is a different and easy thing to skip once a plausible-sounding explanation has already been written down.

The general lesson here: Phase 9's adversarial testing was thorough for what it tested, but it still wasn't a substitute for a real person playing a real, complete match against another real person, over a real network, with real latency. Both kinds of testing found genuine bugs the other one missed.
