# Risk Register

**Status:** Updated after Phase 8 (Secure Development) and Phase 11 (Deployment) — controls implemented and verified for R2–R4, R6, R8–R10; R5 verified live post-deployment; R7 remains an accepted risk
**Scope:** `server/server.js` as of the Phase 2/3 networking implementation, plus findings from the Phase 7 STRIDE pass
**Note:** R1–R7 came out of the initial Phase 5 pass. R8–R10 turned up specifically because STRIDE was applied systematically in Phase 7 — none of the three would have been caught just by asking "can the client cheat?" That's the main argument I have for doing a structured pass instead of trusting an ad hoc review to be enough.

| R1 | Unbounded room creation — rooms are never deleted, so a script joining/disconnecting thousands of unique room IDs grows server memory indefinitely | Availability (DoS) | Medium | High | ~~High~~ **Mitigated** | Room state is deleted once both player slots are empty (see `socket.on('disconnect', ...)`) — verified by scripted test: 5 abandoned rooms created, 0 remained in memory afterward | Remediated immediately — see SD-001 in Security Decisions.md. Residual: a max-concurrent-rooms cap would still guard against many *simultaneously active* rooms; not yet implemented |
| R2 | No rate limiting on `input` or `join` events — a client can flood either at will | Availability (DoS) | Medium | Medium | ~~Medium~~ **Mitigated** | Fixed-window per-socket rate limiting: max 40 `input`/sec, max 5 `join`/10s — verified by scripted test sending 100 rapid input events with no crash or slowdown | Residual: window is fixed per-socket in memory; a distributed flood across many sockets isn't addressed by this alone (see R10) |
| R3 | Player slot takeover on reconnect — a brief disconnect frees the slot for any other client to claim, with no identity/session check | Integrity / Authentication | Medium | Medium | ~~Medium~~ **Mitigated** | Server issues a per-slot reconnection token on join; a 15s grace period pauses (not ends) the match on disconnect, and the same token reclaims the same slot with position/score intact — verified by scripted test (disconnect → reconnect with token → same slot, `gameActive` resumes `true`) | Residual: token lives only in server memory and the client's JS variable — lost on a full page reload, which is an accepted trade-off for this prototype |
| R4 | No input validation on `left`/`right`/`jump` payload types | Integrity | Low | Low | ~~Low~~ **Mitigated** | Server explicitly coerces each field to a strict boolean (`=== true`) before use — verified by scripted test sending non-boolean payloads with no adverse effect | None outstanding |
| R5 | Unrestricted CORS (`origin: '*'`) | Confidentiality / Integrity | Low (prototype stage) | Medium (once deployed publicly) | ~~Low–Medium~~ **Mitigated** | Origin is read from an `ALLOWED_ORIGIN` environment variable (defaults to `*` for local dev), applied to both Express `cors` and Socket.IO's CORS config. `PORT` is also now environment-driven (was hardcoded to 3000, which would have silently broken every target hosting platform) | Remediated at Phase 11: `ALLOWED_ORIGIN` set to the real deployed domain; no longer a configuration gap |
| R6 | No `roomId` validation (length/charset) | Integrity | Low | Low | ~~Low~~ **Mitigated** | `roomId` is validated against `/^[A-Za-z0-9_-]{1,32}$/` before use — invalid codes are rejected with an explicit error, verified by scripted test | None outstanding |
| R7 | Automated/bot input — a script can send frame-perfect input the server cannot distinguish from a human | Integrity (fair play) | High | Low | **Low (accepted)** | None — inherent to any server that trusts client-originated input events | Out of scope to fully solve; remains an accepted risk unless bot-detection becomes a project goal |
| R8 | No persistent, timestamped logging of match/connection events — only ephemeral `console.log`, lost on restart | Repudiation | Low | Low | ~~Low~~ **Mitigated** | Structured JSON-line logging (`server/logs/events.log`) records connect/join/reconnect/disconnect/match events with timestamps, survives a process restart — verified by scripted test | Residual: this is a flat file, not a queryable audit store; fine at current scale, worth revisiting if this ever needs to answer "what happened" questions at volume |
| R9 | Default Express error handler may return a stack trace to the client on an unhandled exception if `NODE_ENV` isn't `production` | Information Disclosure | Low | Low | ~~Low~~ **Mitigated** | A custom Express error-handling middleware returns a generic JSON message regardless of `NODE_ENV`; `x-powered-by` header disabled | None outstanding |
| R10 | No timeout for sockets that connect but never send `join` — each still holds a connection handle indefinitely | Availability (DoS) | Medium | Low | ~~Low–Medium~~ **Mitigated** | Sockets that haven't called `join` within 20s are force-disconnected server-side | None outstanding |
| R11 | `socket.emit('join', null)` crashed the entire server process — any single client could take down the match for every player in every room, no auth or special access required | Availability (DoS) — Critical | High | Critical | ~~Critical~~ **Mitigated** | Null-safe payload handling plus `try`/`catch` around every Socket.IO event handler; process-level `uncaughtException` safety net added as defense-in-depth — verified by scripted attack: repeated malformed `join` payloads (`null`, `undefined`, number, string, array) no longer crash the process, and a fresh client can still join normally afterward | Remediated immediately — see SD-002. | R12 | Re-joining a different room without disconnecting from the first left a permanently occupied "ghost" slot in the original room, bypassing the R1 cleanup fix entirely | Availability (DoS) | Medium | High | ~~High~~ **Mitigated** | `join` now frees any previous room's slot before assigning a new one, exactly as a disconnect would — verified by scripted test: 3 sequential joins from one socket without disconnecting left 0 ghost rooms behind (previously left 2) | Remediated immediately — see SD-003. This defeated R1's original fix via a different path than the one it was tested against |

## What's already well-mitigated (for contrast)

Recording this on purpose, because a risk register that only lists problems undersells the design decisions that were already made:

- **Position, physics, collisions, timer, and score are 100% server-authoritative.** The client cannot spoof any of these regardless of what it sends over the socket (Phase 4 deliverable, verified in `updatePlayer()` and the state broadcast — the client never transmits its own position or score).
- **No client-supplied state is ever trusted as fact.** The server only ever accepts three boolean-shaped inputs (`left`, `right`, `jump`) and computes everything else itself.

## Next steps

Phases 7 (STRIDE), 8 (Secure Development), 9 (Security Testing), and 11 (Deployment) are complete. Of twelve identified risks: eleven are mitigated and verified (eight from Phase 8's own controls, two — R11 and R12 — found and fixed during Phase 9's adversarial testing, and R5 confirmed once `ALLOWED_ORIGIN` was set for the live deployment), and one (R7) is a deliberately accepted risk. See `Security Testing.md` for the full adversarial test methodology and results, and `Security Decisions.md` for SD-002/SD-003 covering why R11 and R12 were remediated immediately rather than deferred.
